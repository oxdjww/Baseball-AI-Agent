# 2026-04-05 개발 로그

## 주요 작업

선취점 알림이 역전 메시지로 발송되던 버그 수정. 코드 경로 추적으로 2차 결함(동점 후 재역전)까지 발견하여 함께 수정.

---

### 1. 선취점 알림 버그 수정 (핵심)

#### 문제

0:0에서 첫 득점 시 `formatFirstScore()`가 아닌 `formatLeadChange()`가 호출되어 역전 메시지가 발송됨.

#### 원인

`LeadChangeNotifier`의 분기 설계 결함.
- 첫 폴링(0:0)에서 Redis에 `"NONE"` 저장 후 return
- 다음 폴링(1:0)에서 `rawPrev = "NONE"` (null이 아님) → `rawPrev == null` 분기를 건너뜀
- `currLeader != prevLeader` → `formatLeadChange()` 경로로 진입

`formatFirstScore()`는 첫 폴링 시점에 이미 득점이 있는 드문 경우에만 호출되고 있었음.

#### 수정

`rawPrev != null` 분기 내에 `isFirstScore` 조건 추가:

```java
boolean isFirstScore = "NONE".equals(prevLeader)
        && Math.min(info.getHomeScore(), info.getAwayScore()) == 0;
```

|------|------|
| `game/alert/LeadChangeNotifier.java` | `isFirstScore` 분기 추가, 로그 메시지 선취점/역전 구분 |
| `LeadChangeNotifierTest.java` | 3개 테스트 추가 |

---

### 2. 동점 후 재역전 오분류 방지 (2차 결함)

#### 문제

`"NONE".equals(prevLeader)` 만으로 선취점을 판별하면, 동점 후 재역전(2:2 → 3:2)도 선취점으로 오분류됨.

#### 수정

`Math.min(homeScore, awayScore) == 0` 조건 추가.
- 야구 점수는 감소하지 않으므로, min이 0이면 이전 동점이 반드시 0:0
- min이 0보다 크면 이전 동점이 n:n (n>0) → 역전 메시지 발송

---

### 3. 재배포 안전성 검증

Redis 키(`game:leader:{gameId}`, TTL 24h)는 앱 재시작과 무관하게 유지되므로 경기 중 재배포 시 중복 발송 없음 확인.

| 시나리오 | 결과 |
|----------|------|
| 경기 중 앱 재배포 (Redis 유지) | 중복 없음 |
| 경기 중 재배포 + 그 사이 점수 변동 | 정상 1회 알림 |
| Redis까지 재시작 (극단적) | 오발송 가능 (기존 한계) |

기존 8개 + 신규 3개 = 11개 테스트, 전체 통과.
