### 4. SSH 터널링 + AI Agent 원격 통합 관제 시스템 구축

**핵심 기술**: SSH, Docker Logs, OpenClaw Cron, AI Agent, Telegram API

**아키텍처**:
```
[Spring Boot - 서버]          [OpenClaw - MacBook(로컬)]
MonitoringAspect              cron: 5분 주기
  └─ 예외 감지                  └─ ssh kbaseball
  └─ Telegram → Admin           └─ docker logs --since 5m
                                └─ ERROR 감지 시 Telegram → Admin
```

**이중 감시의 진짜 가치 (서로 보완하는 역할)**:

| 구분 | 방식 | 역할 |
|------|------|------|
| 1차 (실시간) | Spring Boot MonitoringAspect | 특정 API 예외 즉각 포착 — 전술적 대응 |
| 2차 (5분 주기) | OpenClaw SSH 외부 감시 | 앱 먹통 / DB 연결 완전 단절 등 리스너 자체가 동작 불가한 최악의 상황 감지 — 전략적 대응 |

**성과**:
- 1GB RAM 서버 환경에서 AI 분석 부하를 MacBook으로 분산
- Spring Boot 프로세스가 죽어도 외부 SSH 감시가 동작하는 2중 안전망
- 서비스 가동률(Availability) 극대화

**구축 내역**:
- `~/.ssh/config`에 `kbaseball` 호스트 등록 (id_ed25519 무인증)
- OpenClaw LaunchAgent 상시 실행 (port 18789), 모델: google/gemini-2.5-flash
- cron `37a24adf` 등록: every 5m, isolated session, 에러 감지 시에만 Telegram 알림 (정상이면 무음)

### 5. GlobalExceptionHandler 추가 — 관제 감시 범위 확장

**문제**: `MonitoringAspect`(AOP)가 `*Service` 클래스만 감시 → 컨트롤러에서 직접 던진 예외 누락
- 예: `GameAnalysisTestController.orElseThrow()`, `RainfallTestController` 직접 예외 등

**해결**: `@RestControllerAdvice(annotations = RestController.class)` 추가
- `@RestController` 빈에서 발생한 미처리 예외를 `MonitoringErrorEvent`로 연결
- `@Controller`(Admin HTML) 동작 변경 없음 — 기존 Spring 기본 에러 처리 유지
- **이중 발화 방지**: 예외 스택트레이스에 `MonitoringAspect` 클래스명이 포함되어 있으면 AOP가 이미 처리한 서비스 레이어 예외 → 재발행 생략

**감시 범위 비교**:

| 예외 발생 위치 | 기존 | 변경 후 |
|---|---|---|
| `*Service` 메서드 | ✅ AOP 감지 | ✅ AOP 감지 (동일) |
| `@RestController` 직접 throw | ❌ 누락 | ✅ ControllerAdvice 감지 |
| `@Controller` (Admin HTML) | ❌ 누락 | ❌ 유지 (HTML 동작 보호) |

**파일**: `com.kbank.kbaseball.monitoring.GlobalExceptionHandler`
**테스트**: `GlobalExceptionHandlerTest` — 유닛 테스트 6개 전체 통과
