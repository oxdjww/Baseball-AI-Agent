### 1. 사고 분석: LG vs KIA 경기 우천취소 알림 미발송

#### 현상

2026-04-09 LG vs KIA 경기가 우천취소되었으나, T-3h/T-1h 강수량 알림에서 "비 걱정 없어요!" 메시지가 발송됨.

#### 원인 1: KMA API 관측소 번호 오류

| 팀 | 오류 번호 | 올바른 번호 | 비고 |
|----|-----------|-------------|------|
| HH(한화) | 156 (광주) | 133 (대전) | 홈구장: 한화생명이글스파크 |
| SS(삼성) | 224 (불명) | 143 (대구) | 홈구장: 삼성 라이온즈 파크 |

LG 경기는 잠실(서울 108번)로 관측소가 맞았으나, 당일 실제 강수량이 0mm여서 다른 이유로 취소된 것으로 판단.

#### 원인 2: KMA API의 근본적 한계

`kma_sfctm2.php`는 **예보가 아닌 실측값**을 반환한다. T-3h/T-1h 시점에 비가 안 왔더라도 경기 시작 시각에 비가 올 수 있으며, 이를 사전에 감지할 수 없다.

#### 원인 3: KmaWeatherClient null 응답 NPE

API 응답이 null일 때 방어 코드 없음 → `Double.parseDouble(null)` NPE 가능.

### 2. 시스템 재설계: 우천취소 실시간 알림

기존 설계(T-3h/T-1h 시점에 취소 여부도 함께 확인)의 근본 한계를 인정하고, 독립적인 폴링 스케줄러로 분리.

#### notification_history 테이블 (V7 마이그레이션)

```sql
CREATE TABLE notification_history (
    id                BIGSERIAL    PRIMARY KEY,
    game_id           VARCHAR(64)  NOT NULL,
    notification_type VARCHAR(32)  NOT NULL,
    sent_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_notification_history_game_type
        UNIQUE (game_id, notification_type)
);
```

- unique constraint 기반 원자적 중복 방지
- 서버 재시작/배포 후에도 발송 이력 유지

#### RainCancelPollingScheduler (신규)

- `@Scheduled(cron = "0 0/5 10-20 * * *", zone = "Asia/Seoul")` — 10시~21시 5분 주기
- 취소 알림 SSOT: 강수량 알림(RainAlertTasklet)과 완전 분리
- `isCanceled` → `tryMarkCancelSent()` → 단 한 번만 발송 (race condition 대응)
- 경기 목록 조회 실패 시 관리자 텔레그램 알림, 개별 경기 실패는 로그 후 계속 처리

#### NotificationHistoryService 원자적 삽입

JdbcTemplate 직접 INSERT 사용 (JPA `repository.save()` 불가):
- `DataIntegrityViolationException` 발생 시 JPA EntityManager가 트랜잭션을 rollback-only로 마킹
- `REQUIRES_NEW` 격리와 결합하면 `UnexpectedRollbackException` 발생
- JdbcTemplate은 EntityManager를 우회하므로 예외 catch 후 정상 커밋 가능

#### RainAlertTasklet 단순화

- 취소 감지/발송 로직 완전 제거
- 발송 전 `isCancelAlreadySent()` DB 확인 → 취소 이력 있으면 강수량 알림 생략
- 강수량 정보 전달에만 집중

### 5. 트러블슈팅: H2 + REQUIRES_NEW + DataIntegrityViolationException

**시도 1**: `repository.save()` + `REQUIRES_NEW` + `DataIntegrityViolationException` catch
→ JPA spec상 EntityManager가 트랜잭션을 rollback-only로 마킹 → `UnexpectedRollbackException`

**시도 2**: `@Modifying` + `INSERT ... ON CONFLICT DO NOTHING` native query
→ H2 2.3.232는 `ON CONFLICT DO NOTHING` 미지원 → `JdbcSQLSyntaxErrorException`

**최종 해결**: `JdbcTemplate.update()` 직접 INSERT
→ EntityManager 완전 우회 → 예외 catch 후 `REQUIRES_NEW` 트랜잭션 정상 커밋
